# 유지보수 가이드

콘텐츠가 쌓이기 시작하면 처음엔 안 보이던 문제들이 드러납니다. 이 문서는 "글이 늘어날수록
무엇을 신경 써야 하는가"를 다룹니다. 지금 당장 할 일은 없지만, 아래 기준에 도달하면 이 문서로
돌아와 해당 섹션을 실행하세요.

## 1. 글 100개까지 확장 전략

이 사이트는 DB 없이 파일시스템 + 정적 생성(SSG)이라, 글이 늘어나도 "서버가 느려지는" 문제는
없습니다. 대신 아래 3가지 UX/구조 문제가 순서대로 나타납니다.

### 글 30~50개 시점 — 카테고리 목록 페이지 페이지네이션

`src/app/{category}/page.tsx`는 현재 해당 카테고리의 **모든 글을 한 페이지에** 그리드로 렌더링합니다
(`getAllPostsMeta(category)`를 그대로 `.map()`). 카테고리당 글이 30개를 넘으면 스크롤이 지나치게
길어집니다. 이 시점에 페이지네이션 또는 "더 보기" 버튼을 추가하세요.

- 가장 간단한 방법: `?page=` 검색 파라미터를 받아 `.slice((page-1)*20, page*20)`로 20개씩 자르기
  (`/learn`은 이미 `?topic=` 파라미터를 쓰고 있으니 같은 패턴을 재사용하면 됩니다).
- 목록이 20개 이하인 카테고리는 그대로 둬도 무방합니다. 카테고리별로 필요한 시점이 다릅니다.

### 글 50개 시점 — 홈페이지 "최신 글" 및 카테고리 카드 로직 재확인

홈(`src/app/page.tsx`)의 `getLatestPosts(8, [...])`, 투자 인사이트의 `.slice(0, 3)`처럼
개수를 고정 슬라이스하는 부분은 글이 몇 개든 상관없이 정상 동작합니다(안전). 다만 이 시점부터는
"최신 글"이 특정 카테고리에 편중되지 않는지 눈으로 점검하세요. 한 카테고리만 자주 발행하면
`getLatestPosts`가 그 카테고리로 도배될 수 있습니다 — 이 경우 카테고리별로 최소 1개씩 섞어 넣는
로직으로 바꾸는 것을 고려하세요.

### 글 100개 시점 — 태그 정리(택소노미 정비)

`tags`를 자유 입력으로 운영하면 100개 근처에서 비슷한 태그가 여러 형태로 흩어집니다
(`금리` / `기준금리` / `금리인상` 등). 이러면 [관련 글 추천 알고리즘](./internal-linking.md)의
정확도가 떨어집니다. 100개 근처에서 한 번, 전체 `content/**/*.mdx`의 `tags`를 훑어 동의어를
하나로 통일하는 "태그 정리" 작업을 권장합니다. (모든 `.mdx`가 텍스트 파일이므로 에디터의
전체 찾아바꾸기로 충분합니다.)

## 2. 성능 유지 방법

- **빌드 시간**: `generateStaticParams`가 글 개수만큼 페이지를 정적 생성합니다. 글 수백 개
  수준까지는 Vercel 빌드 시간에 큰 영향이 없습니다. 빌드가 눈에 띄게 느려지면(수 분 이상)
  `npm run build`를 로컬에서 돌려 어느 라우트가 느린지 로그로 확인하세요.
- **이미지**: 현재 `Thumbnail` 컴포넌트는 일반 `<img>` 태그를 사용합니다(외부 URL도 자유롭게
  쓸 수 있도록 의도적으로 `next/image`를 쓰지 않았습니다). 자체 호스팅 이미지가 많아지면
  업로드 전에 압축하고, 가능하면 WebP로 변환해 저장하세요. 이미지가 대부분 자체 호스팅
  (`public/images/...`)으로 바뀌는 시점에는 `next/image` 전환을 검토해도 좋습니다(자동 리사이징/최적화).
- **폰트**: `Noto Sans KR` variable 폰트 1종만 로드하고 있습니다. 폰트를 추가하면 초기 로딩이
  느려지므로, 정말 필요할 때만 추가하세요.
- **의존성**: `next-mdx-remote`, `gray-matter` 외에 무거운 라이브러리를 추가하기 전에
  꼭 필요한지 재검토하세요. 이 프로젝트는 의도적으로 의존성을 최소화했습니다.

## 3. SEO 유지 방법

- **사이트맵 자동화**: `src/app/sitemap.ts`는 `content/` 폴더를 매 빌드마다 다시 읽으므로,
  글을 추가/삭제하기만 하면 사이트맵이 자동으로 최신 상태를 반영합니다. 별도 갱신 작업이 필요 없습니다.
- **Search Console 정기 점검**: 월 1회 정도 Search Console의 **페이지(색인)** 리포트를 확인해
  "색인 생성되지 않음" 페이지가 늘어나는지 점검하세요. 늘어난다면 해당 글의 `description`이
  너무 짧거나 다른 글과 내용이 겹치는지(중복 콘텐츠) 확인합니다.
- **콘텐츠 신선도**: `terms`/`learn`처럼 평가성(evergreen) 글은 시간이 지나 내용이 달라지면
  (세율 변경, 기준금리 정책 변경 등) frontmatter의 `date`를 갱신하세요. 이 값이 그대로
  `ArticleJsonLd`의 `dateModified`와 사이트맵 `lastModified`에 들어가, 검색엔진에 "최신화된
  콘텐츠"라는 신호를 줍니다.
- **끊어진 링크 점검**: 본문에서 다른 글로 링크를 걸었는데([internal-linking.md](./internal-linking.md)
  참고) 나중에 그 글의 슬러그(파일명)를 바꾸면 링크가 깨집니다. **슬러그는 한 번 발행하면 바꾸지
  않는 것**을 원칙으로 하세요. 제목만 바꾸고 싶다면 frontmatter `title`만 수정하고 파일명은
  유지합니다.
- **중복 타이틀/키워드 점검**: [content-strategy.md](./content-strategy.md)의 20개 리스트를
  다 쓴 뒤 새 글감을 잡을 때, 기존 글과 타겟 키워드가 겹치지 않는지 확인하세요. 같은 키워드를
  두 글이 각각 타겟팅하면 서로의 순위를 깎아먹는 "키워드 카니발라이제이션"이 발생합니다.
